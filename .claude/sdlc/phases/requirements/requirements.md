## 3. Содержание

### Область MVP

MVP покрывает CRUD-минимум по задаче плюс фильтр по статусу.
Требования зафиксированы, изменений в пределах текущей итерации не ожидается.

### Список фич

| ID | Фича | Описание | Критерий «готово» |
|---|---|---|---|
| F-01 | Добавление задачи | Пользователь вводит текст и создаёт задачу. | Задача появляется в списке с исходным текстом и статусом «активна». |
| F-02 | Просмотр списка задач | Пользователь видит все задачи проекта. | Список отображает все задачи; порядок детерминированный. |
| F-03 | Отметка выполненной | Пользователь меняет статус задачи на «выполнена». | Статус переключается; изменение видно в списке сразу. |
| F-04 | Удаление задачи | Пользователь удаляет задачу навсегда. | Задача исчезает из списка; повторно не появляется. |
| F-05 | Фильтр по статусу | Пользователь выбирает: активные, выполненные, все. | Список обновляется; счётчик соответствует фильтру. |

### Стейкхолдер и его интересы

- Автор как единственный пользователь-разработчик.
- Интересы: простота использования, ценность SDLC-прогона, own-your-data.

### Границы MVP (не-требования)

- Приоритеты и дедлайны задач — вне MVP.
- Теги, категории, проекты — вне MVP.
- Совместный доступ — вне MVP.
- Облачная синхронизация — вне MVP.

### Неявные качественные атрибуты

- Отклик интерфейса должен быть мгновенным для списка до сотни задач.
- Потеря данных при сбое приложения недопустима.
- Отсутствие внешних зависимостей вне стандартного стека выбранной платформы.
