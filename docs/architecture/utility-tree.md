# Utility Tree -- Todo List

> Приоритизация quality attribute scenarios по методологии ADD 3.0.
> Формат приоритета: **(бизнес-важность, техническая сложность)** --
> H (High), M (Medium), L (Low).

## Статус

Предварительная версия. Quality attribute scenarios ещё не уточнены
через Architecture Review. Ниже -- начальная оценка на основании
контекста проекта (оффлайн SPA, один пользователь, нет развития).

---

## Usability

| ID   | Scenario                                                        | Приоритет |
|------|-----------------------------------------------------------------|-----------|
| QA-U1 | Пользователь создает задачу за 2 действия (ввод текста + Enter) | (H, L)   |
| QA-U2 | Фильтрация и поиск обновляют список мгновенно (< 100ms)       | (M, L)   |

## Performance

| ID    | Scenario                                                          | Приоритет |
|-------|-------------------------------------------------------------------|-----------|
| QA-P1 | Приложение загружается и становится интерактивным за < 2 секунды  | (M, L)   |
| QA-P2 | Список из 500 задач рендерится и фильтруется без лагов (< 200ms) | (L, L)   |

## Modifiability

| ID    | Scenario                                                             | Приоритет |
|-------|----------------------------------------------------------------------|-----------|
| QA-M1 | Добавление нового поля задачи требует изменений в <= 3 модулях       | (M, M)   |
| QA-M2 | Замена localStorage на другой storage не затрагивает бизнес-логику    | (L, M)   |

## Reliability

| ID    | Scenario                                                            | Приоритет |
|-------|---------------------------------------------------------------------|-----------|
| QA-R1 | Данные не теряются при перезагрузке страницы                        | (H, L)   |
| QA-R2 | При ошибке сериализации/десериализации приложение показывает fallback| (M, M)   |

## Testability

| ID    | Scenario                                                       | Приоритет |
|-------|----------------------------------------------------------------|-----------|
| QA-T1 | Бизнес-логика тестируется без DOM и без localStorage           | (M, M)   |

---

## Priority Matrix

> **(H,H)** -- наивысший приоритет, адресовать в первой итерации.
> **(H,M)/(M,H)** -- высокий приоритет. **(H,L)/(L,H)** и **(M,M)** -- средний.

| Приоритет    | Scenarios          |
|--------------|--------------------|
| **(H, L)**   | QA-U1, QA-R1       |
| **(M, M)**   | QA-M1, QA-R2, QA-T1|
| **(M, L)**   | QA-U2, QA-P1       |
| **(L, M)**   | QA-M2              |
| **(L, L)**   | QA-P2              |

## Выводы

Для данного проекта критичны **usability** (простота взаимодействия)
и **reliability** (сохранность данных в localStorage). Performance
и modifiability имеют низкий приоритет из-за малого масштаба данных
и отсутствия планов развития.
