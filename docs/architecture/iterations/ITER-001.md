reference architecture для системы Todo List. Это первая итерация, поэтому
фокус -- на общей структуре: какие контейнеры составляют систему, как они
взаимодействуют, какова роль каждого.

### Драйверы в фокусе

| ID      | Тип        | Название                                       | Роль в итерации |
|---------|------------|------------------------------------------------|-----------------|
| CRN-001 | Concern    | React + бэкенд при оффлайн-режиме             | Основной -- разрешается в этой итерации |
| CON-001 | Constraint | Только оффлайн, данные в localStorage          | Ограничивает пространство решений |
| CON-002 | Constraint | Без аутентификации, однопользовательский режим  | Ограничивает пространство решений |
| QA-T1   | QA         | Бизнес-логика тестируется без DOM/localStorage  | Влияет на внутреннюю декомпозицию |
| QA-M2   | QA         | Замена storage не затрагивает бизнес-логику     | Влияет на границы между слоями |

## Step 4: Choose Design Concepts

### 4.1 Пространство решений

CRN-001 определяет четыре варианта архитектуры. Ниже -- систематическое
сравнение каждого варианта по критериям, важным для production development.

### 4.2 Таблица сравнения вариантов

| Критерий | 1. SPA без бэкенда | 2. Бэкенд как static server | 3. Бэкенд с API + SQLite | 4. Гибрид (API + localStorage cache) |
|----------|--------------------|-----------------------------|--------------------------|--------------------------------------|
| **Effort** | Низкий. Один пакет, Vite build, deploy как статика. | Средний. Два пакета (React + Express), но Express -- тривиальный (< 20 строк). | Высокий. Полноценный API, схема БД, миграции, серверная валидация, тесты на два уровня. | Высокий. Всё из варианта 3, плюс синхронизация cache, conflict resolution. |
| **CON-001** | Полностью соответствует. Данные в localStorage, сервер не нужен. | Соответствует. Static server не касается данных. | **Нарушает**: данные на сервере (SQLite), не в localStorage. Требуется пересмотр CON-001. | Частично нарушает: primary storage на сервере, localStorage -- лишь cache. |
| **CON-002** | Соответствует. Нет сервера -- нет аутентификации. | Соответствует. Static server не имеет пользовательских данных. | Формально соответствует (один пользователь), но серверный API без аутентификации -- потенциальная точка атаки при смене контекста. | То же, что вариант 3. |
| **QA-T1** | Достижима. Бизнес-логику можно вынести в pure-функции, тестировать unit-тестами. | Так же, как вариант 1. Frontend не меняется. | Достижима, но тесты дублируются: серверные (API/DB) + клиентские. | Максимальная сложность тестирования из-за двух storage и синхронизации. |
| **QA-M2** | Достижима через абстракцию storage (repository pattern). | Так же, как вариант 1. | Достижима на сервере (repository over SQLite). На клиенте storage не нужен. | Сложно: нужна абстракция и на клиенте, и на сервере, плюс абстракция sync layer. |
| **Testability** | Высокая. Один runtime (browser), одна кодовая база. | Высокая. Тесты те же, сервер тривиален. | Средняя. Два runtime, интеграционные тесты API, e2e. | Низкая. Три аспекта: API, cache, sync. |
| **Modifiability** | Высокая. Одна точка изменения. | Высокая. Сервер меняется редко. | Средняя. Изменение модели требует миграции БД + API + клиент. | Низкая. Каждое изменение затрагивает 4 слоя. |
| **Risk** | Низкий. Проверенный паттерн (SPA + localStorage). | Низкий. Минимальное усложнение. | Средний. Пересмотр ограничений меняет scope проекта. | Высокий. Overengineering для MVP без планов развития. |
| **Deployment** | Тривиальный: статические файлы (CDN, GitHub Pages, файловая система). | Простой: `node server.js`, один процесс. | Средний: Node.js + SQLite файл, нужен persistent storage. | Сложный: то же, что вариант 3, плюс обработка edge cases при offline. |
| **Соответствие контексту** | Полностью. Greenfield MVP, нет развития, один пользователь. | Приемлемо, но Express не несет ценности. | Не соответствует: overengineering, нарушает CON-001. | Не соответствует: максимальный overengineering для данного контекста. |

### 4.3 Анализ

**Варианты 3 и 4 отклоняются:**

- Вариант 3 нарушает принятое ограничение CON-001 (данные в localStorage).
  Пересмотр ограничения не обоснован -- нет бизнес-причин для серверного
  хранения. Дополнительно: SQLite + миграции + серверная валидация создают
  значительный overhead для однопользовательского MVP.
- Вариант 4 -- суперпозиция проблем вариантов 1 и 3. Синхронизация между
  двумя storage -- решённая в индустрии задача, но решённая для другого
  контекста (распределённые системы, multi-device). Для локального MVP
  это architectural mismatch.

**Выбор между вариантами 1 и 2:**

Оба варианта полностью соответствуют ограничениям и одинаковы с точки зрения
frontend-архитектуры. Разница -- наличие Express-сервера, который раздаёт
статику.

Аргументы за вариант 2 (static server):

- Пользователь указал "React с бэкендом" -- наличие сервера формально
  соответствует запросу.
- Production development предполагает серверный процесс для раздачи приложения
  (не открытие HTML-файла из файловой системы).
- Static server может выполнять полезные функции: gzip, SPA fallback
  (history API), security headers, health check endpoint.
- Добавляет менее 20 строк кода к проекту, не усложняет тестирование.
- Естественная точка расширения, если в будущем потребуется серверная логика
  (вопреки текущим планам).

Аргументы за вариант 1 (без бэкенда):

- Максимальная простота. Нет лишнего процесса.
- Deployment -- копирование файлов в любое место.
- Express-сервер, который лишь раздаёт статику -- фактически дублирует
  функциональность nginx/Apache/Caddy/GitHub Pages.

### 4.4 Рекомендация

**Рекомендуется вариант 2 -- "Backend as Static Server".**

Обоснование:

1. **Production development** -- проект разрабатывается командой для production.
   Наличие серверного процесса -- стандартная практика для production SPA.
   Это обеспечивает единый способ запуска (`npm start`), health check, корректную
   обработку SPA-маршрутизации (history API fallback), и базовые security headers.

2. **Соответствие стеку** -- пользователь указал "React с бэкендом". Вариант 2
   честно включает бэкенд-процесс в архитектуру, при этом не усложняя систему.
   Это -- minimal viable backend.

3. **Все ограничения соблюдены** -- CON-001 (localStorage), CON-002 (без auth).
   Бэкенд не касается данных пользователя.

4. **Quality attributes** -- QA-T1 и QA-M2 достижимы так же, как в варианте 1.
   Frontend-архитектура идентична: бизнес-логика в pure-функциях, storage за
   абстракцией.

5. **Минимальный overhead** -- Express static server это ~15-20 строк кода.
   Не усложняет тестирование, не создаёт дополнительных зависимостей
   (Express -- единственная серверная зависимость).

6. **Deployment** -- `node server.js` (или `npm start`) -- единый артефакт,
   один процесс. Проще чем "собрать статику и настроить отдельный веб-сервер".

### 4.5 Предварительная структура контейнеров

На основании выбранного варианта 2, система todoApp декомпозируется на:

| Контейнер     | Технология   | Ответственность                                 |
|---------------|--------------|--------------------------------------------------|
| **frontend**  | React + Vite | SPA: UI, бизнес-логика, state management, persistence (localStorage) |
| **server**    | Node.js + Express | Раздача собранного SPA, SPA fallback, security headers |

Взаимодействия:

- `user -> server` -- HTTP-запросы (GET для загрузки SPA)
- `server -> frontend` -- раздаёт статические файлы (HTML/JS/CSS)
- `frontend -> browser` -- чтение/запись в localStorage API

Это предварительная схема. Финальная версия будет зафиксирована в C4-модели
и ADR после утверждения варианта (Step 5-6).

## Step 5: Instantiate Architectural Elements

### 5.1 Тип структуры

**B) C&C structure** (Component & Connector) -- определяются runtime-элементы,
их ответственности, интерфейсы и взаимодействия.

### 5.2 Элементы

Система todoApp декомпозирована на два контейнера:

| Контейнер | Технология | Ответственность |
|-----------|------------|-----------------|
| **frontend** | React + Vite (TypeScript) | SPA: UI, бизнес-логика, state management, persistence (localStorage) |
| **server** | Node.js + Express | Раздача статических файлов SPA, SPA fallback, security headers, gzip |

### 5.3 Ответственности и интерфейсы

Полная Element Responsibility Table: [`docs/architecture/views/containers-ert.md`](../views/containers-ert.md)

Ключевые характеристики:

**frontend (React SPA):**
- Содержит **всю** бизнес-логику приложения (CRUD задач, валидация, фильтрация)
- Управляет состоянием приложения (React state)
- Обеспечивает персистентность через localStorage API
- Потребляет: localStorage API, History API (браузер)
- Ограничения по QA: бизнес-логика в pure-функциях (QA-T1), storage за абстракцией (QA-M2)

**server (Express Static Server):**
- Раздаёт собранные файлы из `dist/`
- SPA fallback: все non-file запросы возвращают `index.html`
- Security headers: X-Content-Type-Options, X-Frame-Options
- **Не** содержит бизнес-логики, **не** хранит данных, **не** предоставляет API

### 5.4 Взаимодействия (Connectors)

| Источник | Цель | Протокол | Тип | Описание |
|----------|------|----------|-----|----------|
| user | server | HTTP GET | sync | Браузер загружает страницу приложения |
| server | frontend | HTTP Response | sync | Сервер отдаёт статические файлы (HTML/JS/CSS) |
| frontend | browser | localStorage API | sync | SPA читает/записывает данные задач |
| user | frontend | DOM Events | sync | Пользователь взаимодействует с UI |

### 7.2 Tactics Conflict Check

Проверка конфликтов между архитектурными тактиками:

| Тактика 1 | Тактика 2 | Конфликт? | Анализ |
|-----------|-----------|-----------|--------|
| Static server (минимальный бэкенд) | localStorage persistence | Нет | Server не касается данных. Контейнеры ортогональны. |
| SPA fallback (server) | Client-side routing (frontend) | Нет | Взаимодополняющие тактики: server обеспечивает fallback, frontend обрабатывает маршруты. |
| Pure business logic (QA-T1) | Storage abstraction (QA-M2) | Нет | Обе тактики направлены на separation of concerns и взаимно усиливают друг друга. |
| Gzip compression (server) | SPA bundle (frontend) | Нет | Compression прозрачна для frontend. |

**Конфликты не обнаружены.** Выбранные тактики ортогональны или комплементарны.
