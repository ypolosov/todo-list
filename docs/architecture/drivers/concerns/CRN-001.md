# CRN-001: React + бэкенд при оффлайн-режиме

## Description

Пользователь указал стек "React с бэкендом", при этом приложение работает
полностью оффлайн с хранением данных в localStorage. Это создает
архитектурное противоречие: классический бэкенд (API-сервер с базой данных)
не имеет смысла при оффлайн-режиме с клиентским хранилищем.

## Impact

Неразрешенный вопрос блокирует принятие архитектурных решений:

- Структура проекта (monorepo vs отдельные репозитории)
- Выбор инструментов сборки и запуска
- Определение границ ответственности frontend/backend
- Стратегия деплоя

## Possible Resolutions

1. **SPA без бэкенда** -- React-приложение собирается статически (Vite/CRA),
   раздается через любой статический сервер или открывается как файл.
   Вся логика на клиенте. Бэкенд отсутствует.

2. **Бэкенд как статический сервер** -- Node.js/Express сервер, который
   лишь раздает собранный React-бандл. Вся бизнес-логика остается
   на клиенте в localStorage. Бэкенд не имеет API-эндпоинтов.

3. **Бэкенд с API, но данные в файле/SQLite** -- Пересмотр ограничения
   CON-001: вместо localStorage использовать серверное хранение
   (файл, SQLite). Приложение остается локальным, но данные хранятся
   на сервере. Работает оффлайн (localhost).

4. **Гибрид** -- Бэкенд предоставляет API, но клиент кэширует данные
   в localStorage. При оффлайн-режиме работает с кэшем.
   Избыточно для MVP без планов развития.
